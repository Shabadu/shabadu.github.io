This is the standard git branching and merging model. Please note that while this is the standard, variations or alterations may be called for in certain projects. If alterations are made, those alterations should be fully documented in the project's documentation and communicated to both the client and to the entire project team.

Initialization

  • A new repository is created with a master branch. A stage branch is then created from master. These are the only two lasting branches.
  • One Issue per branch
  • All commit messages must contain the issue number. This allows us to track work and will automatically associate a work item to a commit.

Master Branch

  • The master branch should reflect currently "released" code.
  • It should be locked down with write privileges being given solely to the project's designated architect, lead developer and build master.
  • The master branch is updated by a merge from the stage branch when preparing for a release.
  • The branch should be tagged when a release is performed [release].[sprint].[hotfix].
    • Example: 0.6.1

Stage Branch

  • As each feature passes QA, the entire feature branch will be merged into the stage branch.
  • The pull request to stage should be opened by a lead or build master with the original developer added as a reviewer to validate all of the changes.

Sprint Branch

  • Sprint branches are created from stage at the beginning of each sprint to provide a baseline branch for sprint work.
  • Name the sprint branch with the sprint number.
    • Example: sprints/sprint-2

Develop Branch

  • A new develop branch is created at the start of each sprint at the same time as the sprint branch.
  • As tickets are completed, pull requests are created to merge code into the develop branch for QA.

Feature Branches

  • Created off the baseline sprint branch
  • Name with issue number and in the features folder
  • If one feature branch depends on another, branch from them. Never branch from develop
